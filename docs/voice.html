CHAPTER D

AI 語音控制子系統(Mode 10)

語音指令辨識管線之設計與實作

D.1系統設計

採 Push-to-Talk 操作模型:按住左肩鈕錄音、放開即觸發辨識管線。音訊以 16 kHz / 16-bit 單聲道取樣,直接寫入 PSRAM 緩衝(上限 10 秒,約 320 KB),結束後組裝 44-byte WAV 標頭,經 TLS 上傳雲端。

圖 D-1 Mode 10 語音指令辨識管線(端到端延遲約 3 秒)

D.2指令集定義

LLM 以系統提示詞 + few-shot 範例約束輸出為固定結構 JSON(response_format: json_object,temperature 0.1):

action參數對應語音範例
takeoff—「起飛」「飛起來」
land—「降落」「下來」
movedirection(6 向)、duration_sec(1–10)「向前飛三秒」
stop—「停下來」「懸停」
modemode(0 / 1 / 2)「切到 GPS 模式」
unknown—無法解析之輸入(安全預設)

D.3工程處理要點

  • 文字前處理:Whisper 輸出常含結尾標點(「起飛。」),與 few-shot 範例字面失配導致解析失敗;解析前統一去除前後空白與中英文標點(問題 I-06)。
  • 強健性設計:錄音振幅低於閾值不上傳(省去無效請求);WiFi 失聯自動重連;各階段失敗均於 TFT 顯示具體錯誤(HTTP 狀態碼 / 失敗階段),且任一狀態下可立即重新按鈕錄音,不會卡死。
  • 服務選型歷程:曾評估以多模態模型(音訊直入)單次完成 STT + 解析,延遲較低且只需單一服務;但免費層級速率限制(15–30 請求/分鐘)無法支撐連續整合測試(問題 I-05),最終採兩段式管線將負載分散於不同服務之獨立額度。此經驗直接促成「本地離線辨識」之中期規劃。
  • 音訊雜訊抑制:擴大機 shutdown 腳於開機即拉低,消除 I²S 時脈雜訊被放大之問題(問題 I-04)。
  • 憑證管理:WiFi 帳密與 API 金鑰隔離於 secrets.h,排除於版本控制之外。
頻段共存說明:WiFi 與 nRF24 同處 2.4 GHz。Mode 10 測試程式運作時不啟用 nRF24;正式整合時將採分時策略 —— 語音辨識期間暫停控制鏈路屬不可接受之安全風險,故規劃改以本地離線辨識(ESP-SR)從根本移除 WiFi 依賴。
