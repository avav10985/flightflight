CHAPTER D

AI 語音控制子系統(Mode 10)

語音指令辨識管線之設計與實作

D.1系統設計

採 Push-to-Talk 操作模型:按住左肩鈕錄音、放開即觸發辨識管線。音訊以 16 kHz / 16-bit 單聲道取樣,直接寫入 PSRAM 緩衝(上限 10 秒,約 320 KB),結束後組裝 44-byte WAV 標頭,經 TLS 上傳雲端。

圖 D-1 Mode 10 語音指令辨識管線(端到端延遲約 3 秒)

D.2指令集定義

LLM 以系統提示詞 + few-shot 範例約束輸出為固定結構 JSON(response_format: json_object,temperature 0.1):

action參數對應語音範例
takeoff—「起飛」「飛起來」
land—「降落」「下來」
movedirection(6 向)、duration_sec(1–10)「向前飛三秒」
stop—「停下來」「懸停」
modemode(0 / 1 / 2)「切到 GPS 模式」
unknown—無法解析之輸入(安全預設)

D.3工程處理要點

  • 文字前處理:Whisper 輸出常含結尾標點(「起飛。」),與 few-shot 範例字面失配導致解析失敗;解析前統一去除前後空白與中英文標點(問題 I-06)。
  • 強健性設計:錄音振幅低於閾值不上傳(省去無效請求);WiFi 失聯自動重連;各階段失敗均於 TFT 顯示具體錯誤(HTTP 狀態碼 / 失敗階段),且任一狀態下可立即重新按鈕錄音,不會卡死。
  • 服務選型歷程:曾評估以多模態模型(音訊直入)單次完成 STT + 解析,延遲較低且只需單一服務;但免費層級速率限制(15–30 請求/分鐘)無法支撐連續整合測試(問題 I-05),最終採兩段式管線將負載分散於不同服務之獨立額度。此經驗直接促成「本地離線辨識」之中期規劃。
  • 音訊雜訊抑制:擴大機 shutdown 腳於開機即拉低,消除 I²S 時脈雜訊被放大之問題(問題 I-04)。
  • 憑證管理:WiFi 帳密與 API 金鑰隔離於 secrets.h,排除於版本控制之外。
頻段共存:WiFi 工作於 ≤2.472 GHz,nRF24 控制鏈路選用 CH 88(2.488 GHz)—— 頻率錯開使兩射頻可同時運作。整合實測:語音辨識期間 50 Hz 控制鏈路不中斷。

D.4主程式整合(雙核心架構)

語音管線已整合進地面站主程式,關鍵為雙核心任務分工:

核心任務說明
核心 1(主迴圈)50 Hz 飛行控制搖桿取樣、nRF24 收發、TFT 顯示、PTT 偵測 —— 不含任何阻塞呼叫
核心 0(語音任務)語音管線I²S 錄音、WiFi TLS 上傳、STT 與 LLM 解析 —— 阻塞數秒亦不影響控制

跨核心通訊採「單寫者」共享變數:主迴圈寫 PTT 與模式狀態,語音任務寫解析結果與狀態字串,以 fresh 旗標交接,無鎖設計。I²S 採全雙工配置 —— 麥克風(RX)與擴大機(TX)共用 BCLK / WS 時脈線。

D.4.1 語音回應(closed-loop 確認)

指令解析完成後,喇叭以合成語音回覆確認(「收到,起飛」「轉一號馬達」「聽不懂,請再說一次」)。因回應句為固定集合,採預錄方案:以語音合成(zh-TW)離線產生 8 句 WAV 存於 SD 卡,辨識完成即時播放 —— 零額外延遲、不占 API 額度、離線可用。

D.4.2 安全映射:單馬達展示模式

於姿態控制完全調校前,語音指令採保守映射(編譯期旗標切換):「起飛」「轉 N 號馬達」→ 指定單顆馬達以低 PWM 輕轉 3 秒後自動停止;飛行控制器端僅於未武裝且油門收底時受理,武裝後自動失效。完整飛行映射(虛擬搖桿:起飛 / 懸停 / 移動 / 降落,含搖桿即時搶回機制)程式已完成,待飛行調校成熟後以單一旗標啟用。
