📖 開發故事(點標題展開)

兩個月、192 個提交。最精彩的不是功能,是每一個「為什麼不會動」最後變成「原來如此」的過程。

2026 / 04 中

▶ 專案啟動:決定一切自己來

不買現成飛控板,從 ESP32 + MPU6050 開始打造。

硬體軟體
  • 目標:理解四軸飛行器的每一層 —— 從感測器讀值到馬達輸出
  • 選 ESP32 而非 Arduino:雙核心、夠快的浮點運算跑 PID、之後還能上 WiFi
  • 機架、馬達、電調自己配,電路自己畫
2026 / 05 中

▶ 飛控三階段:Gyro → Manual → PID

把飛控拆成三個獨立程式逐步驗證,每一階段都能單獨測試。

軟體
  • Gyro 階段:先確認 MPU6050 讀值正確、互補濾波算出的姿態角可信
  • Manual 階段:搖桿直接控馬達,驗證通訊鏈路與馬達混控
  • PID 階段:三軸 PID 閉環,飛機自己保持水平
  • 5/25 PID 調參完成:軸對齊、水平基準校正、搖桿觸發校正流程
2026 / 05 末

▶ 感測器大擴充 + 雙向遙測

GPS、ESP32-CAM 即時影像、氣壓計;手把不只發指令,還收得到飛機回報。

硬體軟體
  • GPS 模組加衛星訊號強度(SNR)監看,藍牙輸出 debug
  • ESP32-CAM:第一人稱視角即時影像,可在飛行中調畫質
  • NRF24 改雙向:飛機每秒回傳姿態、高度、航向、GPS 座標
  • 9 種飛行模式編碼定案:SW_A 十位 + SW_B 個位(00/01/02/10/11/12/20/21/22)
2026 / 06 初

▶ 手把 V2-A:跳過 V1 直上 ESP32-S3 + 彩色螢幕

原本規劃的字元 LCD 版直接跳過,改用 ESP32-S3 + 2.4 吋彩色 TFT。

硬體
  • 三板架構:A 板(主控)、B 板(螢幕)、C 板(被動元件樞紐)
  • 接線總表反覆迭代十幾版:TFT 從並列改 SPI 省 9 隻腳、SD 改獨立模組
  • 油門搖桿拆彈簧(不回中)、肩鍵當 yaw、GPIO 對調讓走線整齊
  • TFT 中文介面:即時顯示姿態、高度、航向、模式、PID 參數選單
2026 / 06 / 09-10

▶ 整合除錯週:SPI 搶 bus 與功率尖峰

所有模組裝在一起後,各種「單獨都好、合體就掛」的問題輪番上陣。

踩坑
  • SD 模組搶 SPI bus:有 level shifter 的 SD 模組不放開 MISO 線,NRF24 讀不到資料 → 對照測試鎖定後先停用 SD
  • NRF24 功率尖峰:PA_MAX 發射瞬間電流讓 LDO 電壓下垂 → 降到 PA_MIN 先確保穩定
  • PID 太兇:輕推搖桿馬達狂轉 → 參數保守化 + 最大傾角限制
  • 順手做了手機頭追原型:手機陀螺儀 → WiFi → 飛機上兩顆伺服馬達轉鏡頭
2026 / 06 / 11

▶ 電源之戰:WiFi 一開就重啟的真兇

Mode 10 需要 WiFi,但 ESP32-S3 一執行 WiFi 初始化就無限重啟 —— 用對照實驗抓到真兇。

踩坑硬體
  • 症狀:連 5 行的最簡 WiFi 程式都會讓晶片瞬間斷電重啟(brown-out)
  • 對照實驗:同一條 USB 線、同一支程式 —— 經典 ESP32 完全正常,S3 模組必掛 → 真兇是 S3 板上的弱 LDO(約 300mA),撐不住 WiFi 啟動的 400mA 尖峰
  • 解法:換 1A LDO 的開發板 + 外掛 LD1117 建第二條 3.3V 軌
  • 雙軌架構:板上 LDO 只餵晶片、LD1117 專餵周邊(TFT/NRF24/麥克風),WiFi 尖峰再也不影響螢幕
  • 定下供電鐵律:USB 與電池不可同時接(兩顆 LDO 並聯會反灌)
2026 / 06 / 11

▶ Mode 10:對手把說「起飛」

按住肩鈕說話 → 雲端語音辨識 → AI 解析成飛行指令 → 螢幕顯示結果。全程約 3 秒。

AI軟體踩坑
  • 流程:INMP441 錄音存 PSRAM → 組 WAV → WiFi 上傳 Groq Whisper(語音→文字)→ Llama 70B(文字→JSON 指令)→ TFT 顯示
  • 指令集:起飛 / 降落 / 移動(方向+秒數)/ 停止 / 切模式
  • 免費額度之戰:試過 Gemini 一次完成辨識+解析(更快),但連續測試撞 429 限流 → 退回 Groq 雙服務分流
  • 細節坑:Whisper 回的文字帶句號讓 AI 解析失敗 → 先清標點再送;喇叭開機雜音 → 擴大機預設靜音
  • 因為 USB 與電池互斥、測試時看不到電腦訊息 → 所有狀態與錯誤都顯示在 TFT 上
