1.摘要

本專案自零開始設計並實作一套完整的四軸無人機系統,不採用任何現成飛控板或遙控器,涵蓋三個子系統:

  • 飛行控制器:以 ESP32 為核心,自行實作慣性量測融合(互補濾波)、三軸串級 PID 姿態控制、X 型四馬達混控,並整合 GPS、氣壓計、磁力計與雷射測距感測器。
  • 地面站(手把):以 ESP32-S3 為核心,2.4 吋彩色 TFT 全中文操作介面,雙搖桿 + 9 種飛行模式,經 nRF24L01+ 與飛行控制器進行 50 Hz 雙向通訊(指令下行/遙測上行)。
  • AI 語音控制子系統(Mode 10):地面站內建 I²S 麥克風,按住肩鈕錄音後經 WiFi 上傳雲端,以 Whisper 模型進行中文語音辨識,再由大型語言模型解析為結構化飛行指令(JSON),全程延遲約 3 秒。

開發過程採對照實驗方法定位多項系統級問題,包括穩壓器容量不足導致之 WiFi 欠壓重啟、SPI 匯流排佔用衝突、射頻發射電流尖峰等,並完成雙穩壓器電源架構之重新設計。本文件記錄系統之完整技術規格、設計決策與驗證結果。

4.硬體設計

4.1 地面站電源架構

初版設計中所有 3.3 V 負載均由 ESP32-S3 模組板載穩壓器供電。實測發現該穩壓器(額定約 300 mA)無法承受 WiFi 射頻啟動之瞬時電流(約 400 mA),導致欠壓重啟(brown-out)。重新設計後採雙穩壓器分軌架構:晶片與周邊各自擁有獨立 3.3 V 軌,僅共地,射頻電流尖峰不再影響周邊供電品質。

圖 4-1 地面站雙穩壓器分軌電源架構

本架構之直接效益:WiFi 啟動與發射期間,周邊軌電壓不受晶片軌負載變動影響,TFT 顯示器閃爍與 nRF24 通訊不穩定問題隨之消除。操作程序上,韌體燒錄時僅接 USB(電池斷開),飛行與整合測試時僅接電池(USB 斷開);此限制同時意味序列埠監看與周邊(TFT / 麥克風)無法同時使用,故所有執行期狀態與錯誤訊息均設計為顯示於 TFT。

5.通訊協定

控制鏈路採 nRF24L01+ Enhanced ShockBurst,通道 88(2.488 GHz),自訂 5-byte 管線位址(避開函式庫常見預設值,降低同頻干擾)。地面站以約 50 Hz 發送指令封包;遙測資料以 ACK payload 機制搭載於確認封包回傳,不佔用額外傳輸時隙。

5.1 下行封包 Signal(地面站 → 飛行控制器,12 bytes)

欄位型別說明

modeuint8飛行模式兩位數編碼(§2.3)
flagsuint8bit 0:完整校準觸發;其餘保留
paramIDuint8線上調參:1–5 = Kp/Ki/Kd/Kp_yaw/Ki_yaw;100/101/102 = 快速校準/完整校準/PID 存檔

封包大小 20 bytes,符合 nRF24 ACK payload 上限(32 bytes)並保留擴充空間。失聯保護:地面站訊號中斷超過 1 秒,飛行控制器立即將油門歸零。

6.飛行控制軟體

6.1 姿態估計 — 互補濾波

陀螺儀積分短期精確但長期漂移;加速度計長期穩定但受振動干擾。互補濾波以高通 / 低通互補方式融合兩者:

roll₀ = 0.98 × (roll + gyro_roll × dt) + 0.02 × acc_roll pitch₀ = 0.98 × (pitch + gyro_pitch × dt) + 0.02 × acc_pitch

偏航軸無重力參考,採相同結構但以磁力計絕對航向作為 2% 修正源,抑制陀螺積分漂移;無磁力計時退化為純積分模式。

6.2 姿態控制 — 角度環 PID

搖桿值線性映射為目標角度(最大傾角 ±20°,經實飛測試由 ±15° 上調):

setpoint = (stick − 127) × (MAX_ANGLE / 127) // MAX_ANGLE = 20° output = Kp·e + Ki·∫e·dt + Kd·de/dt // e = setpoint − 量測角

6.3 X 型四馬達混控

M1(前左 CW) = T + pitchOut + rollOut − yawOut M2(前右 CCW) = T + pitchOut − rollOut + yawOut M3(後左 CCW) = T − pitchOut + rollOut + yawOut M4(後右 CW) = T − pitchOut − rollOut − yawOut 各輸出限幅於 [1000, 2000]

6.4 校準系統

  • 快速校準:靜置取陀螺零偏;完整校準:另含水平基準(加速度計姿態零點)。
  • 校準期間執行動作偵測:量測值波動超過閾值即判定校準失敗、拒絕寫入,防止手持晃動產生錯誤基準。
  • 校準結果與 PID 參數均存於 ESP32 NVS(非揮發性儲存),斷電不遺失;兩種校準均可由地面站選單遠端觸發。

6.5 開發驗證策略

飛控程式採三階段獨立驗證:Gyro(僅驗證感測器讀值與姿態解算)→ Manual(搖桿直通馬達,驗證通訊與混控)→ PID(閉環自穩)。每階段為獨立可燒錄程式,問題定位範圍明確;此外另維護 28 支單元測試程式(各感測器、TFT、I²S、WiFi 最小化測試),系統級問題均可快速降階重現。

7.AI 語音控制子系統(Mode 10)

採 Push-to-Talk 操作模型:按住左肩鈕錄音、放開即觸發辨識管線。音訊以 16 kHz / 16-bit 單聲道取樣,直接寫入 PSRAM 環形緩衝(上限 10 秒,約 320 KB),結束後組裝 44-byte WAV 標頭,經 TLS 上傳雲端。

7.1 指令集定義

LLM 以系統提示詞 + few-shot 範例約束輸出為固定結構 JSON(response_format: json_object,temperature 0.1):

action參數對應語音範例
takeoff—「起飛」「飛起來」
land—「降落」「下來」
movedirection(6 向)、duration_sec(1–10)「向前飛三秒」
stop—「停下來」「懸停」
modemode(0 / 1 / 2)「切到 GPS 模式」
unknown—無法解析之輸入(安全預設)

7.2 工程處理要點

  • 前處理:Whisper 輸出常含結尾標點(「起飛。」),解析前先去除前後空白與中英文標點,避免與 few-shot 範例失配。
  • 強健性:錄音振幅低於閾值不上傳;WiFi 失聯自動重連;各階段失敗均於 TFT 顯示具體錯誤(HTTP 狀態碼 / 階段名),且任一狀態下可立即重新錄音。
  • 服務選型:曾評估以多模態模型單次完成 STT + 解析(延遲較低),但免費層級速率限制(15–30 RPM)無法支撐連續測試,最終採兩段式管線將負載分散於不同服務之獨立額度。
  • 音訊雜訊:擴大機 shutdown 腳於開機即拉低,消除 I²S 時脈雜訊被放大之問題;憑證(WiFi / API 金鑰)隔離於 secrets.h 並排除於版本控制之外。
