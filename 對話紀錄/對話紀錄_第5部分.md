我用104陶瓷電容還是100P陶瓷電容


### 🤖 Claude `2026-05-24 17:36:38`

**用 104**，不要用 100P。

## 兩個容量差 1000 倍

| 標示 | 換算 | 用途 |
|------|------|------|
| **104** | 10 × 10⁴ pF = **100,000 pF = 100 nF = 0.1µF** | ✅ **電源去耦**（你要的）|
| **100P / 101** | 100 pF = 0.0001µF | ❌ 太小，給 RF 高頻電路用 |

差距：**1000 倍**。100P 對電源完全沒用。

## 為什麼是 0.1µF（104）

電源去耦要「雙電容並聯」：
- **100µF 電解**（C1）：吸收**低頻**大波動（馬達啟動、電池電壓掉）
- **0.1µF 陶瓷（104）**：吸收**高頻**雜訊（MCU 切換、PWM 高頻噪音）

100pF 太小，**高頻吸收能力可以忽略不計**。

## 怎麼快速辨認 104

```
陶瓷電容上印 3 個數字：
   前兩位 = 數值
   第三位 = 後面加幾個 0（單位 pF）

104 → 10 + 4 個 0 → 100,000 pF → 0.1µF
103 → 10 + 3 個 0 → 10,000 pF → 0.01µF（也常見）
102 → 10 + 2 個 0 → 1,000 pF
101 → 10 + 1 個 0 → 100 pF
100 → 10 + 0 個 0 → 10 pF（裸的「100」也是這個意思）
```

買的時候找標 **「104」**或寫**「0.1µF」**或**「100nF」**，三種都是同一顆。

## 2. 接線（Arduino Nano）

| OLED 腳 | Nano |
|---------|------|
| VCC | 5V |
| GND | GND |
| SDA | **A4** |
| SCL | **A5** |

> A4 / A5 是 Nano 的硬體 I²C 預設腳，你目前 TX 程式沒用到（搖桿在 A0~A3），**完全不衝突**。

## 3. 雙向通訊：用 NRF24 的 ACK Payload

NRF24 內建功能：**每次 TX 送資料 → RX 端可以回傳 32 bytes 的 ACK payload**。

```
地面端：radio.write(&command, sizeof(command));
         ↓ 自動同時讀取
         if (radio.isAckPayloadAvailable()) {
           radio.read(&telemetry, sizeof(telemetry));  // 拿到遙測
         }

飛機端：if (radio.available()) {
         radio.writeAckPayload(1, &telemetry, sizeof(telemetry));  // 預先準備好 ACK
         radio.read(&command, sizeof(command));
         // 下次主控 write 過來時，會自動送出剛剛存的 telemetry
       }
```

**優點：**
- 一次 RF 來回完成「下指令 + 收回應」
- 不需要切換 TX / RX 模式（NRF24 自動處理）
- 延遲低（< 5ms）

**唯一變更：** 兩端 `radio.setAutoAck(true)` + `radio.enableAckPayload()`

## 4. 設計遙測資料結構

每次 ACK 帶 32 bytes，能塞很多東西：

```cpp
struct Telemetry {
  int16_t  roll;          // 角度 × 10（節省 byte）
  int16_t  pitch;
  int16_t  yawRate;
  uint16_t altitude_cm;   // 高度 cm
  uint16_t battery_mV;    // 電壓 mV
  uint8_t  gps_sat;       // GPS 衛星數
  uint8_t  status;        // bit0=armed, bit1=lowBat, bit2=gpsFix
  int32_t  gps_lat_e7;    // 經緯度 × 10^7
  int32_t  gps_lon_e7;
  uint16_t dist_front;    // 雷射前方 mm
  uint16_t dist_back;
  uint8_t  rx_strength;   // 訊號強度（可選）
};   // 約 22 bytes < 32 OK
```

```cpp
void drawScreen() {
  u8g2.clearBuffer();
  u8g2.setFont(u8g2_font_6x10_tf);

  u8g2.setCursor(0, 8);
  u8g2.print(tele.status & 0x01 ? "ARM " : "OFF ");
  u8g2.setCursor(40, 8);
  u8g2.printf("Bat: %.2fV", tele.battery_mV / 1000.0);

  u8g2.setCursor(0, 20);
  u8g2.printf("R:%5.1f P:%5.1f",
              tele.roll / 10.0, tele.pitch / 10.0);

  u8g2.setCursor(0, 32);
  u8g2.printf("Alt:%4.1fm  Sat:%d",
              tele.altitude_cm / 100.0, tele.gps_sat);

  // ... 更多欄位 ...

  u8g2.sendBuffer();
}

void loop() {
  // ... 原本送搖桿資料 ...
  if (radio.isAckPayloadAvailable()) {
    radio.read(&tele, sizeof(tele));
  }
  drawScreen();
  delay(50);   // 20Hz 刷新
}
```

## 8. 額外考量

- **Nano RAM 緊張**：U8g2 用 page mode（每次只渲染一條），不要用 full buffer
- **OLED 更新率 20Hz 夠用**：飛機資料變不快
- **遙測延遲 < 50ms**：用 ACK payload 不會卡住操控
- **背光**：OLED 自發光，**陽光下還看得到**（不像 LCD）

## 標準 X 型四軸轉向

**從上方往下看（俯視）：**

```
            前方
       
   M1 ◯─────◯ M2     M1: 前左 → CW（順時針）
   CW         CCW     M2: 前右 → CCW（逆時針）
      ╲    ╱
        ╳
      ╱    ╲
   M3 ◯─────◯ M4     M3: 後左 → CCW（逆時針）
   CCW         CW     M4: 後右 → CW（順時針）

            後方
```

**規律：** 對角線兩顆同方向（M1+M4 都 CW、M2+M3 都 CCW）。

## 為什麼要這樣

每顆螺旋槳轉動會對機身產生**反扭矩**（牛頓第三定律）。
- CW 槳 → 對機身產生 CCW 扭矩
- CCW 槳 → 對機身產生 CW 扭矩

如果 4 顆同方向 → 反扭矩全部疊加 → 飛機原地狂轉沒辦法飛。

**對角線同向 + 相鄰反向** → 反扭矩互相抵消 → 機身穩定。要 yaw 旋轉時，故意讓一對加速、一對減速 → 反扭矩失衡 → 飛機旋轉。

## 怎麼測試轉向

**不裝螺旋槳**，用程式（你目前可以控油門）：

1. 推油門 → 4 顆同時轉
2. 從**飛機正上方往下看**每顆馬達
3. 看馬達**外殼（鐘罩）**的轉動方向
4. 對照上面的圖表

**目視重點：** 看不太出來的話可以在馬達邊緣黏一小段膠帶當記號，慢速轉動就看得清楚。

## 轉向錯了怎麼辦

很簡單，**ESC 跟馬達之間有 3 條線**（無刷馬達三相），**任意對調 2 條** → 方向就反過來。

```
ESC ─── 線A ────┐         ESC ─── 線A ─┐ ╲
              ├ 馬達       ╳            │  ╲ 對調 2 條
ESC ─── 線B ──┤             ESC ─── 線B ┤  ╱
              │            ╳            │ ╱
ESC ─── 線C ──┘            ESC ─── 線C ─┘
              （順時針）          （逆時針）
```

**任 2 條對調都行**（B↔C、A↔B、A↔C 都可）。

## 螺旋槳也要對

CW 馬達 → 裝 **CW 槳**（標 "CW" 或 "P"，逆紋）
CCW 馬達 → 裝 **CCW 槳**（標 "CCW" 或 "R"，正紋）

裝反 → 不會產生升力，反而把飛機推下去。

通常 4 顆螺旋槳一組會有 **2 個 CW + 2 個 CCW**，標記在槳上。

## 測試步驟（建議順序）

```
1. ☐ 確認 M1 在前左、M2 前右、M3 後左、M4 後右

5. ☐ 依方向裝對應的螺旋槳（CW 槳給 CW 馬達）

6. ☐ ⚠️ 螺旋槳裝好後不要再通電測試，PID 沒寫不會自穩
```

## 為什麼一般沒事

LOLIN D32 板上設計：
```
USB 接頭 ──→ 二極體 ──┐
                      ├──→ USB 腳（5V 共用軌） ──→ AMS1117 ──→ 3.3V
ESC BEC ─────────────┘
```

- USB 與 BEC 共享 **同一條 5V 軌**
- 二極體擋住「USB 電倒灌回 ESC BEC」
- 兩個 5V 電壓差小（5.0V vs 5.0V）→ 自然分流，**不會打架**

> 跟之前說的「**4 顆 BEC 並聯**」不同：那是 4 個主動穩壓器互鬥；現在是「**USB 被動電 + 1 顆 BEC**」，USB 端只有二極體不會還手，所以安全。

## 但「最安全」的測試做法

**測試 PID 時建議：USB 給 ESP32、電池只給 ESC（拔掉 ESC BEC 進 ESP32 的那條紅線）**

```
USB (電腦) ───→ LOLIN D32 USB 腳 ───→ ESP32 + 序列埠
                                              ↓
                                        4 條訊號出去
                                              ↓
電池 → PDB → 11.1V 動力 → ESC × 4 ──── 馬達

⚠ ESC1 紅線（BEC）→ 暫時拔掉、包起來
   ESC 黑線（GND）→ 必須留著（共地）
```

### 為什麼推薦這做法

| 項目 | 雙電源（你目前）| USB Only |
|------|--------------|---------|
| 馬達轉 | ✅ | ✅（電池供動力）|
| ESP32 工作 | ✅ | ✅（USB 供電）|
| 序列埠 | ✅ | ✅ |
| ESC 起動瞬間 ESP32 重置 | 可能會 | 不會 |
| 馬達啟動電壓崩潰 | 可能影響 ESP32 | 完全隔開 |

馬達啟動瞬間 BEC 會輕微壓降，雙電源時可能讓 ESP32 偶爾 brownout 重置。**USB Only 完全隔離**。

## 第三選項：完全靠電池（最寫實但麻煩）

```
電池 → PDB → ESC → BEC 紅線 → ESP32 USB 腳
                  → 動力給馬達
拔掉 USB 線
```

- 模擬真實飛行狀態
- **但沒有序列埠** → 看不到 PID 數值、無法即時調參
- 只在「程式已調好」階段才用這個

## 我的建議

**現在這階段（調 PID）：**

1. 拔掉 ESC1 紅線（從 ESP32 USB 腳那邊拔，**ESC 那邊先別動**避免短路）
2. 用熱縮套包好
3. USB 接電腦
4. 電池接 PDB（給 ESC 動力 + ESC 黑線共地 ESP32）
5. 序列埠開著調 PID

**未來飛行時：**

1. 拔 USB
2. 把 ESC1 紅線**插回** ESP32 USB 腳
3. 電池一接全部一起開機

兩個模式切換不用改程式，只是物理上接線不同。

## 為什麼**不**建議「USB + BEC + 電池」三個同時

- 雖然 LOLIN 設計上不會壞
- 但**多一個變因，除錯時不知道是哪邊問題**
- 馬達 BEC 雜訊可能干擾 ESP32 USB 通訊
- 安全角度：少接一條線就少一個風險點

簡單說：**測試時 USB Only 是最乾淨的做法**。

三個問題分開答：

## 1. 電解 + 陶瓷電容 → **並聯**

```
VCC ───┬─────┬───→ 給 ESP32 / 模組
       │     │
      C1    C2
     100µF  0.1µF
       │     │
GND ───┴─────┴───
```

兩個並聯在 5V 軌 ↔ GND 之間，吃**不同頻段**的雜訊：
- **C1 100µF** 電解：低頻（馬達啟動 / 電池電壓抖動）
- **C2 0.1µF** 陶瓷（104）：高頻（MCU 切換 / PWM 開關噪音）

兩顆同時並聯才能涵蓋全頻段。**注意 C1 電解有極性，正極接 5V、負極接 GND**。

## 2. 兩條 5V 紅軌怎麼分

| 模組 | 可不可以吃 5V | 為什麼 |
|------|-------------|-------|
| **NRF24L01+PA/LNA** | ❌ **絕對 3.3V** | 沒 LDO，5V 直接燒 |
| LOLIN D32 USB 腳 | ✅ 5V | 板上 AMS1117 降到 3.3V |
| MPU6050 模組（GY-521）| ✅ 5V | 模組有 LDO |
| BMP280 模組 | ✅ 5V（大多）| 看模組標示 |
| TCA9548A 模組 | ✅ 5V | 模組有 LDO |
| VL53L0X 模組 | ⚠️ **看模組** | Adafruit 版 5V OK，裸晶 3.3V |
| GPS NEO-6M | ✅ 5V | 大部分模組 |
| ESP32-CAM | ✅ 5V（必須） | 設計給 5V |
| ESC 訊號線 | 不用接 5V | 訊號線本身是 PWM，不吃電 |

**所以你的「2 條紅都 5V」策略：**

```
紅軌 A（5V）→ ESP32 USB 腳、ESP32-CAM、GPS
紅軌 B（5V）→ MPU6050、BMP280、TCA9548A、VL53L0X（如果你的模組有 LDO）
獨立 3.3V 軌 → 只給 NRF24L01+PA/LNA
```

> NRF24 的 3.3V **從 LOLIN 的「3V」腳取**（LOLIN 板上 AMS1117 從 5V 降下來的）。NRF24 加 100µF 去耦電容也是焊在這條 3.3V 上。

**怎麼確認你的 VL53L0X 模組**：找模組板上有沒有印 "**3.3V-5V**" 字樣，或量 VCC ↔ GND 之間有沒有 SOT-23 包裝（LDO 晶片）。沒 LDO → 只能 3.3V。
