## 主控板：建議 ESP32，不要 Arduino Nano

- **Nano (ATmega328P)**：16MHz、2KB RAM，跑四軸姿態解算 + PID 已經很吃力，**完全無法**處理影像串流或 WiFi。
- **ESP32**：240MHz 雙核、內建 WiFi/藍牙，可以跑 MPU6050 + PID + 遙控。
- 但**單一 ESP32 同時跑飛控 + 影像串流會不穩**（飛控需要穩定 250Hz~1kHz 迴圈，影像會搶 CPU）。

**建議雙 MCU 架構：**
- 飛控：ESP32（或更穩的 STM32，但初學先 ESP32）
- 影像：**ESP32-CAM** 獨立一顆，自己做 WiFi 串流
- 兩者間不需通訊，各做各的

## 陀螺儀：直接選 6 軸或 9 軸，別用 3 軸

- **3 軸（只有 gyro）**：只能測角速度，積分會漂移，無法知道絕對姿態 → 不夠
- **6 軸（gyro + accel）MPU6050**：可用互補濾波/Kalman 求姿態，四軸最低標配，約 NT$60
- **9 軸（加磁力計）MPU9250 / BMI270+BMM150**：多了航向角（yaw）絕對參考，GPS 定點懸停會更穩
- **10 軸（加氣壓計）BMP280**：定高模式必備

**建議**：MPU6050 + BMP280 起步；想做 GPS 懸停再上 9 軸。

## GPS

- **NEO-6M**：便宜（~NT$150）、入門常見，冷啟動慢
- **NEO-M8N**：多星系（GPS+GLONASS+北斗），定位更快更準，推薦
- 注意 GPS 要裝在機架**上方**遠離電機/ESC，避免干擾

## 影像方案

| 方案 | 優點 | 缺點 |
|------|------|------|
| ESP32-CAM + WiFi | 便宜（~NT$200）、直接手機看 | 延遲 200~500ms、距離短（<100m）、FPV 飛行不夠即時 |
| 類比 FPV（5.8GHz 發射器 + 小相機）| 延遲低、距離遠 | 需要 FPV 眼鏡或接收器，不能直接手機看 |
| 數位圖傳（DJI O3 等）| 專業 | 貴，幾千塊起 |

你要「手機即時看」，ESP32-CAM 最直接，但要接受延遲，**不建議拿來當 FPV 操控依據**，只當紀錄/偵察用。

## 其他必要元件

- **電機**：2204 或 2206 無刷馬達 × 4（依機架尺寸）
- **ESC 電調**：無刷電子調速器 × 4（支援 OneShot/DShot 更好）
- **機架**：F330 / F450（初學者建議 450 較穩）
- **螺旋槳**：1045 或 8045，兩正兩反
- **電池**：3S 2200mAh LiPo（11.1V），配 XT60 接頭
- **電源分配板 PDB**：分電 + BEC 給飛控供電
- **遙控**：
  - 簡易：ESP32 + 手機藍牙/WiFi App（延遲高，不建議真飛）
  - 正規：FlySky FS-i6 + FS-iA6B 接收器（~NT$1500，六通道）

## 你該先回答的問題

2. **目標**：純練手做得起來就好，還是要穩定戶外飛行？
3. **遙控**要用手機還是買專用遙控器？（安全性差很多）
4. **影像延遲**你可以接受到什麼程度？

先這幾個方向決定後，我再幫你開完整 BOM 和接線架構。

老實說：**OV7670 + Nano + NRF24L01 傳影像到手機，這條路不建議走**。我拆開說為什麼，然後給你更好的用法。

## OV7670 的問題

- **介面是 8-bit 並列（非 SPI/I2C）**：光是資料腳就要 8 根 + 控制腳 4~5 根，Nano 的腳位幾乎被吃光
- **沒內建 FIFO 的版本**：Nano 根本來不及抓一整張 frame（時脈跟不上）
- **有 FIFO（AL422B）的版本**：可以抓，但 Nano 只有 2KB RAM，一張 QQVGA 160×120 RGB565 = 38KB，**塞不下整張**，只能一小段一小段讀
- **最高現實幀率**：QQVGA 大概 1~2 fps，畫面小到像郵票

## NRF24L01+ 傳影像的頻寬

- 標稱 2Mbps，實際有效約 **1Mbps**
- 每個封包只能 **32 bytes** payload
- 一張 QQVGA 160×120 RGB565 = 38,400 bytes → 1200 個封包
- 理論 ~3 fps，實際加上重傳、CRC、手機端解碼更慢
- **PA+LNA 只增加距離，不增加頻寬**

## 還有：NRF24L01 不能直接連手機

手機沒有 NRF24 接收器。你會需要：
`無人機上 Nano + NRF24 → 地面 Nano + NRF24 → 再經 藍牙/WiFi/USB → 手機`

三段接力，延遲疊加，**根本不是即時畫面**。

## 兩片 Nano 分工可行嗎？

技術上可以（一片抓 OV7670 + NRF24 傳影像，另一片跑飛控），但：
- 兩片 Nano 加起來算力還不如一顆 ESP32
- 重量、佈線、耗電都增加
- 飛控那片 Nano 跑 MPU6050 + PID + GPS + 遙控已經接近極限，**沒空間再加功能**

## 我的建議：善用你有的東西，各司其職

把 NRF24L01+ PA LNA 拿去做它**擅長**的事 —— **遙控鏈路**，不是影像：

```
┌─────────────────────────────────────────┐
│ 無人機端                                │
│ ┌──────────┐    ┌─────────────────┐    │
│ │ ESP32    │    │ ESP32-CAM       │    │
│ │ (飛控)   │    │ (獨立影像)      │    │
│ │ + MPU6050│    │ WiFi → 手機     │    │
│ │ + GPS    │    └─────────────────┘    │
│ │ + NRF24  │                           │
│ └────┬─────┘                           │
└──────┼──────────────────────────────────┘
       │ 2.4GHz NRF24 鏈路（控制+遙測）
       ▼
┌──────────────┐
│ 地面遙控器    │
│ Nano + NRF24 │
│ + 搖桿       │
│ + OLED顯示   │
└──────────────┘
```

**這樣每個零件做它擅長的事：**

| 元件 | 任務 | 為什麼適合 |
|------|------|-----------|
| ESP32（飛控）| MPU6050 + GPS + PID + 電調 PWM | 算力夠，GPIO 夠 |
| NRF24 + PA/LNA | 遙控指令 + 回傳遙測（電量、高度、GPS） | 低延遲（<10ms）、長距離（>500m）、頻寬剛好 |
| ESP32-CAM | 獨立用自己的 WiFi 串流到手機 | 專門為這設計，便宜（~NT$200） |
| Nano（地面端）| 讀搖桿、顯示 OLED、經 NRF24 送指令 | 夠用，Nano 最適合的場景 |

**NRF24 當遙控鏈路的優勢**：
- 延遲低（適合操控）
- PA+LNA 版本射程 500m~1km
- 比你買 FS-i6 遙控器省錢，還能自訂遙測資料

## 結論

- **OV7670 + NRF24 傳影像 = 放棄**，硬做出來也沒實用價值
- **手邊的 NRF24+PA+LNA** → 拿去做**遙控鏈路**，這是它的強項
- **影像** → 花 NT$200 買 ESP32-CAM，獨立 WiFi 給手機
- **飛控** → 一顆 ESP32 處理 MPU6050 + GPS + PID + NRF24 接收

OK，這問題其實是四軸入門者最難的一關：**長距離影像到手機**。先把現實講清楚，再給你三條路。

## 先定義「長距離」

| 距離 | 難度 | 方案 |
|------|------|------|
| <200m | 簡單 | ESP32-CAM 原廠就夠 |
| 200m~1km | 中等 | ESP32-CAM + 外接天線 / 5.8G FPV |
| 1~5km | 困難 | 5.8G 高功率 FPV / 4G 圖傳 |
| >5km | 專業 | DJI 級數位圖傳，幾萬塊 |

你想要的距離是多少？這決定一切。

## 為什麼「手機直接看」會卡住

手機只吃三種無線：**WiFi / 藍牙 / 4G**。所以長距離影像要進手機，只有三條路：

---

### 🅰️ ESP32-CAM + 升級天線（預算 NT$500 內）

- **原廠 PCB 天線**：室外空曠約 100~200m
- **換 IPEX 外接天線 + 空拍機端貼片**：300~500m
- **地面端加高增益八木天線（Yagi）對準空中**：可到 **800m~1.2km**
- 手機直接連無人機的 WiFi 熱點看
- **缺點**：延遲 200~500ms，不能靠這個飛 FPV，只能當「大致看一下」

> 這是最便宜、最快做出來的路，缺點是延遲高。

---

### 🅱️ 5.8GHz 類比 FPV + 手機 OTG 接收器（預算 NT$1500~2500）

典型配置：
```
無人機：小型 FPV 相機（如 Caddx Ant）+ 5.8G VTX（如 Eachine TX805, 25~600mW可調）
地面：ROTG01 / ROTG02 接收器 → USB OTG → Android 手機 App
```

- **射程**：200mW 約 1~2km、600mW 約 3~5km（空曠、線對線）
- **延遲**：<40ms，**可以靠這個 FPV 操控**
- **畫質**：類比 NTSC/PAL，模糊但即時
- **ROTG02 Plus**：~NT$1200，**直接插 Android 手機看**，很成熟
- **iPhone 不行**，Android 限定

> 這是「長距離 + 低延遲 + 手機看」的最佳解，FPV 玩家的標準做法。

⚠️ **台灣法規注意**：5.8GHz VTX 功率超過 25mW 需要審驗，戶外要留意。

---

### 🅲 4G LTE 圖傳（預算 NT$1500~3000）

- 模組：**SIM7600**、**EC200S** 等 4G 模組 + 鏡頭
- 無人機透過 4G 網路把影像推到雲端（RTMP/RTSP 伺服器），手機再從雲端看
- **射程**：只要有訊號就能飛（理論上無限）
- **延遲**：500ms~2s（走網際網路）
- **缺點**：
  - 需要 SIM 卡（物聯網卡或一般上網卡）
  - 耗電大（2~3W）
  - **台灣飛無人機超視距基本上違法**（遙控者要能目視無人機）
  - 訊號死角會斷線

> 技術可行，但法規和用途上比較適合「固定路線巡檢」而非娛樂。

---

## 遙控鏈路也要跟上

**重點**：影像跑到 1km，但遙控只能 300m → 無人機會飛回不來。

你的 **NRF24L01+ PA+LNA 剛好適合**：
- 空曠實測 **800m~1.5km**
- 地面端再加高增益天線可到 **2km+**
- 跟 5.8G FPV 搭起來距離剛好匹配

## 我的推薦（假設你要 500m~1.5km）

```
飛控 MCU:      ESP32
姿態:          MPU6050 + BMP280
GPS:           NEO-M8N
遙控下行:      NRF24L01+ PA/LNA（你已有）
影像：         🅱️ 方案 —— 小型 FPV 相機 + 5.8G VTX 200mW + ROTG02 插手機
```

**為什麼推 5.8G FPV 而不是 ESP32-CAM？**
- 延遲低 20 倍，真的可以靠畫面操控
- 距離輕鬆破 1km
- 抗干擾比 WiFi 好（WiFi 2.4G 會跟 NRF24 打架！）

**順帶解決一個你還沒發現的坑**：
> 如果用 ESP32-CAM（2.4G WiFi）+ NRF24（2.4G）+ 遙控器（2.4G），三個同頻會**互相干擾**。
> 用 5.8G 影像 + 2.4G 遙控，頻段分開，穩定很多。

## AI 避障的現實

你要知道：**DJI Mavic 3 的避障用了 8 顆視覺感測器 + 專用 AI 晶片**，研發花了數十人年。DIY 玩家想做到「即時視覺避障」基本上不可能，主要卡在：

1. **算力**：ESP32 跑不動 YOLO，連 MobileNet 輕量版都吃力
2. **延遲**：影像→AI→決策→電機 整個迴路必須 <50ms，不然撞上了才反應
3. **準確度**：單鏡頭沒深度資訊，AI 只能辨識「前面有東西」但不知道**多遠**
4. **功耗/重量**：能跑視覺 AI 的板子（Jetson Nano、RPi + Coral）都太重太耗電

**結論：NT$3000 預算做視覺 AI 避障 = 不可能**。即使 NT$30000 也很勉強。

## 務實的替代方案

### 方案 A：用**距離感測器**做避障（強烈推薦）

| 感測器 | 射程 | 價格 | 特性 |
|--------|------|------|------|
| HC-SR04 超音波 | 2~4m | NT$40 | 便宜，但戶外風噪干擾大 |
| **VL53L1X ToF 雷射** | 4m | NT$150 | **推薦**，小、輕、準 |
| TFmini-S 雷射 | 12m | NT$900 | 真正能用的避障距離 |

**典型配置**：前 + 左 + 右 + 下（定高） 4 顆 VL53L1X，總花費 NT$600，ESP32 I²C 匯流排直接讀（要加 TCA9548A 多工器，因為 I²C 地址相同）。

這個做法**務實、能飛、能避障**，雖然不是「AI」，但達成的效果接近你要的。

把 AI 從無人機搬到地面：

**優點**：
- 算力沒限制（手機或 PC 都能跑大模型）

- 總迴路延遲 300~500ms（影像上傳 + AI + 指令下傳）
- **這個延遲來不及真正「避障」**，只能做「偵測目標」或「追蹤拍攝」
- 需要穩定的 WiFi 連線

## rx.JPG 電路原理速覽

**這張圖是有刷馬達驅動**（你不會用到，但原理值得懂）。每顆馬達旁邊的 4 個元件：

**① IRFZ44N（N-MOSFET）= 電子開關**
接成「低壓端開關」。Gate 給 HIGH → 馬達通電；Gate 給 LOW → 斷電。

**② 100Ω 閘極電阻 = 保護 Arduino**
MOSFET 的 Gate 是電容，切換瞬間會灌很大電流。加 100Ω 限流到 50mA，不會燒掉 Arduino 的 I/O 腳。

**③ 10kΩ 下拉電阻 = 安全接地**
Arduino 重開機時腳位浮接，浮接電壓可能誤觸發 MOSFET 導通，馬達會亂轉。下拉電阻把 Gate 強制拉到 0V，保證關閉。

**④ 1N4007 飛輪二極體 = 反電動勢保護**
馬達是電感，MOSFET 切斷的瞬間線圈會感應出上百伏高壓（反電動勢）。二極體反向並聯在馬達兩端，正常時不導通；高壓出現時導通把能量洩掉，保護 MOSFET 不被擊穿。

**調速原理（PWM）**：
Arduino 每秒開關 490 次，開關比例（Duty Cycle）決定馬達平均電壓。50% Duty = 半速、100% Duty = 全速。馬達的機械慣性讓它感覺不到這個高頻閃爍。

**為什麼你的無刷馬達不能用這個電路？**
無刷是三相馬達，需要 6 顆 MOSFET + 換相邏輯。那些邏輯已經包在 **ESC** 裡面了，所以你只要給 ESC 一條 PWM 訊號（1000~2000µs）就能控速。

**不是小訊號模型**。這個電路屬於**開關模式（Switching mode）**，也叫**大訊號模型（Large-signal model）**。

## 兩者的根本差異

| 項目 | 小訊號模型 | rx 電路（開關模式） |
|------|-----------|-------------------|
| MOSFET 工作區 | 飽和區（放大區）某個 Q 點附近 | 只在**截止區 ↔ 導通區（三極管區）**之間切換 |
| 訊號幅度 | 很小，對工作點做線性化近似 | 0V ↔ 5V 全擺幅，根本不線性 |
| 分析工具 | 轉移電導 gm、rds、hybrid-π 等效 | Duty Cycle、R<sub>DS(on)</sub>、開關時間 |
| 輸出 | 連續波形（放大） | 方波 → 靠平均值表現「類比」 |
| 重疊原理 | 成立（線性） | 不成立（非線性） |

## 為什麼 MOSFET 在這裡不能當放大器

如果你把 MOSFET 放在飽和區當線性放大器來調馬達速度（這就是真正的「小訊號+工作點」做法），結果會是：

- MOSFET 上會有很大的壓降（例如 V<sub>DS</sub> = 6V）
- 電流同時也大（例如 2A）
- **功耗 = 6V × 2A = 12W** → MOSFET 冒煙
- 效率低得驚人

**開關模式的優勢：**
- MOSFET 導通時：V<sub>DS</sub> ≈ 0.04V（R<sub>DS(on)</sub> × I），功耗極小
- MOSFET 截止時：I = 0，功耗為零
- 只有**切換瞬間**有短暫損耗
- 整體效率可達 **95% 以上**

## 更精確的分類

rx 電路屬於電力電子學裡的幾種相關模型：

1. **開關模型（Switched model）**：MOSFET 理想化成「理想開關」，只有 ON/OFF 兩種狀態
2. **平均模型（Averaged model）**：把高頻 PWM 波形「平均化」，得到一個連續的等效電路。這是分析 DC-DC 轉換器、馬達驅動很常用的手法
3. **狀態空間平均法（State-space averaging）**：更進階的，把 ON 和 OFF 兩個狀態的方程式加權平均

如果你要計算「給定 duty cycle D 下馬達的平均轉速」，用的就是**平均模型**：

$$V_{avg} = D \times V_{bat}, \quad \omega \approx \frac{V_{avg} - I R_m}{K_e}$$

## 什麼時候會出現小訊號模型？

如果你要分析**「PWM duty cycle 微小擾動時，馬達轉速如何響應」**，這時候才會在「平均模型的工作點」上做**小訊號線性化**。例如：

- 設計馬達的速度閉環 PID 時
- 分析控制系統的傳遞函數 H(s)
- 計算頻寬、相位裕度

這個「**在平均模型上做小訊號分析**」是電力電子控制裡的經典手法，但這已經是**控制系統層級**，不是電路層級了。

## 一句話總結

| 層級 | 模型 | 用途 |
|------|------|------|
| 電路級（你看到的 rx 圖）| 開關模式 / 大訊號 | 設計硬體、算功耗、選元件 |
| 系統級（平均化後）| 平均模型 | 算穩態關係、設計控制器 |
| 控制級（平均模型再線性化）| 小訊號模型 | 調 PID、分析穩定性 |

所以：**rx 圖本身是開關模式大訊號電路，不是小訊號模型**。但如果你之後要寫 PID 調速，那個分析過程**會用到**小訊號模型。

## MPU6050 裡有什麼

| 感測器 | 能測什麼 | 能不能算高度 |
|--------|---------|------------|

| 加速度計（Accel） | 三軸加速度（含重力） | 理論可以，實際**不行** |

## 為什麼加速度計推不出高度

理論上：
$$\text{位置} = \iint \text{加速度}\,dt^2$$

實際的致命問題：

**① 二次積分會把雜訊放大成災難**

加速度計的雜訊約 0.01g（≈0.1 m/s²）。看起來很小，但：
- 1 秒後位置誤差：½ × 0.1 × 1² = **5 cm**
- 10 秒後：½ × 0.1 × 100 = **5 m**
- 30 秒後：**45 m**

不用十秒你的無人機就「以為自己飛到樹頂」。

**② 重力分量減不乾淨**

加速度計永遠讀到 1g 重力。你必須先扣掉重力才能得到「真正的加速度」。但飛機只要歪 1°：

$$\text{垂直誤差} = g \times \sin(1°) \approx 0.17 \text{ m/s}^2$$

歪 1° 產生的假加速度就比真實上升加速度還大。

**③ 偏移漂移（Bias Drift）**

溫度變化 10°C，加速度計偏移會飄 0.02g，又是另一個持續累積的誤差源。

**結論：加速度計 + 陀螺儀只能提供「姿態角」（roll/pitch/yaw），無法提供「高度」**。這也是為什麼全世界所有商用四軸都必須配高度感測器。

**方向對，但需要細分**。MOSFET 本質上確實是「電壓控制電流」的元件，但 rx 電路裡的運作比這句話更特殊。

## MOSFET 的本質：電壓控制型

對比兩大類電晶體：

| 元件 | 控制方式 | 控制量 → 被控量 |
|------|---------|----------------|
| BJT（雙極性） | 電流控制 | I<sub>B</sub>（基極電流）→ I<sub>C</sub>（集極電流）|
| **MOSFET** | **電壓控制** | **V<sub>GS</sub>（閘源電壓）→ I<sub>DS</sub>（汲源電流）** |

所以你說「用電壓控制電流」**完全正確**。

MOSFET 的 Gate 跟通道之間隔著一層氧化層（像電容），Gate 理論上**不吃電流**，只靠電壓建立電場來感應出導電通道。

> 這就是為什麼 Arduino 可以輕鬆推動 MOSFET —— 不用提供電流給它，只要給電壓。

## 但在 rx 電路裡，工作模式是「開關」不是「放大」

MOSFET 有三個工作區：

| 工作區 | V<sub>GS</sub> | 特性 | 用途 |
|--------|--------------|------|------|
| 截止區（Cutoff） | < V<sub>th</sub>（~2V）| I<sub>D</sub> = 0 | 開關 OFF |
| 飽和區（Saturation）| 中間 | I<sub>D</sub> ≈ gm × V<sub>GS</sub>（真正「電壓控制電流」）| 放大器 |
| 三極管區（Triode/Ohmic）| > V<sub>th</sub> 很多 | 像電阻 R<sub>DS(on)</sub> | 開關 ON |

rx 電路只在**截止區 ↔ 三極管區**之間跳動，**從來不進入飽和區**。也就是說：

- Arduino 給 5V HIGH → MOSFET 完全導通（三極管區），像一條 0.024Ω 的電線
- Arduino 給 0V LOW → MOSFET 完全截止，像斷路

**這時候電流的大小其實不是 V<sub>GS</sub> 直接決定的**，而是由：

$$I_{motor} = \frac{V_{battery} - V_{EMF}}{R_{motor} + R_{DS(on)}}$$

V<sub>GS</sub> 只決定**開關狀態**，馬達電流大小由電池電壓和馬達本身決定。

## 所以最精確的說法

分成兩個層次：

**① 元件層次**：MOSFET 是電壓控制型元件 ✓（你的直覺正確）

**② 電路層次**：rx 電路用 MOSFET 當「電壓控制的開關」

**③ 系統層次**：透過 PWM 佔空比（也是「平均電壓」）控制馬達的**平均電流**，進而控制轉速

所以整條控制鏈其實是：

```
Arduino 輸出電壓 (PWM)
       ↓（控制開關時序）
MOSFET ON/OFF 切換
       ↓（調變）
馬達兩端平均電壓
       ↓（歐姆定律）
馬達平均電流
       ↓（電磁力）
馬達轉速
```

每一層都是「用前者控制後者」，但每一層的機制不同。

## 為什麼這件事在工程上有意義

- **BJT 當開關**：Arduino 必須持續提供基極電流（可能 5~10mA），一個腳頂多推 5 顆以內
- **MOSFET 當開關**：Arduino 只在切換瞬間吸/灌電流（幾 mA，給 Gate 電容充放電），靜態幾乎不吃電

這就是為什麼**馬達驅動、電源切換、功率電路基本上都用 MOSFET 而不用 BJT**，不是因為它「比較強」，而是因為它**對控制端幾乎零負擔**。

## 一句話總結

> 你說「電壓控制電流」方向對。更精準地說：**Gate 電壓控制通道的開啟 / 關閉，PWM 佔空比控制平均電流。**

| 192 | 75% | 平均 8.3V，高速 |

## 四個角 = 四顆馬達獨立控制

四組一模一樣的電路，各自接到 Arduino 不同的 PWM 腳，就能**獨立調整**四顆馬達轉速 —— 這正是四軸飛行控制的核心（升降、翻滾、偏航都是靠四顆馬達轉速差異達成）。

## 但要記住

這電路只能用於：
- ✅ **有刷 DC 馬達**（兩條線、直接給電就轉）
- ❌ 不能用於你手上的**無刷馬達**（三條線，需要 ESC 換相）

你的無刷馬達 → 直接用 ESC，ESP32 輸出 PWM（1000~2000µs）給 ESC，ESC 自己處理三相換相。原理上相似（都是 PWM 控制），但**你不用自己焊 MOSFET 電路**。

## 物理原理

**1. 升力與推力（牛頓第三定律）**
螺旋槳旋轉向下推動空氣，空氣反作用於螺旋槳產生升力。四顆馬達轉速差異產生扭矩差，控制俯仰、翻滾、偏航。

**2. 角動量守恆**
對角線兩顆螺旋槳順時針、另兩顆逆時針，扭矩互相抵消，避免機身自旋。

**3. PID 控制與慣性感測**
MPU6050 透過科氏力（陀螺儀）與重力分量（加速度計）測量姿態，PID 演算法即時修正馬達輸出達成穩定。

**4. 氣壓高度與 GPS 定位**
BMP280 依據大氣壓隨高度遞減（氣壓公式）推算高度；GPS 以三角定位計算經緯度座標。

## TCA9548A → VL53L1X × 4 分配

| TCA9548A 通道 | 接到 | 安裝位置 |
|---------------|------|---------|
| SC0/SD0 | VL53L1X #1 | 前方 |

## 電源架構

```
3S LiPo 電池 (11.1V)
       │
       ▼
   XT60 接頭
       │
       ▼
┌──────────────┐
│   PDB        │
│  (電源分配板)│
└──────┬───────┘
       │
   ├─→ 11.1V → ESC1 (動力)
   ├─→ 11.1V → ESC2 (動力)
   ├─→ 11.1V → ESC3 (動力)
   ├─→ 11.1V → ESC4 (動力)
   └─→ 5V BEC ─┬─→ ESP32 VIN
              └─→ ESP32-CAM 5V

ESP32 內部 3.3V 穩壓器
       ├─→ MPU6050 VCC
       ├─→ BMP280 VCC
       ├─→ TCA9548A VCC
       ├─→ VL53L1X × 4 VCC（透過 TCA9548A）
       ├─→ NRF24 VCC（旁邊加 100μF 電容！）
       └─→ GPS VCC（NEO-M8N 多數版本支援 3.3~5V）
```

## 電池電壓監測（重要）

ESP32 ADC 最大 3.3V，但 3S 滿電 12.6V，必須加分壓電阻：

```
電池 + ────┬─── 30kΩ ───┬─── 10kΩ ───┬─── GND
           │             │             │
           動力電源      GPIO 34       GND

分壓比 = 10/(30+10) = 1/4
12.6V × 1/4 = 3.15V ✓（在 ADC 範圍內）
```

讀到的 ADC 值 × 4 = 真實電池電壓

## 馬達 + 螺旋槳配置（俯視）

```
      FRONT (前)
   M2 ◯─────◯ M1
   CCW  │     │  CW
        │ ESP32│
        │      │
   CW   │     │  CCW
   M3 ◯─────◯ M4
       BACK (後)

M1 → GPIO 25
M2 → GPIO 26
M3 → GPIO 27
M4 → GPIO 14
```

⚠️ 螺旋槳方向：對角同向、相鄰反向（M1/M3 順時針 CW；M2/M4 逆時針 CCW）

## 關鍵注意事項

1. **NRF24 加電容**：模組 VCC/GND 之間焊一顆 100μF 電解電容，否則切換時電壓抖動會當機
2. **ESP32-CAM 完全獨立**：不接 ESP32，只共用電池（透過 PDB 的 5V BEC），它自己接 WiFi
3. **ESC 訊號線只接訊號 + GND**，紅線（ESC 內建 BEC 5V）**不要接 ESP32**，避免電源衝突
4. **GPS 天線朝上**，遠離 ESP32 和馬達（避免電磁干擾）
5. **MPU6050 裝在機架中心**，越接近重心越準，並用海綿減震
6. **所有 GND 共地**：電池負極 → PDB GND → ESP32 GND → 所有模組 GND，**必須連通**

## MPU6050 vs MPU6500 規格比較

| 項目 | MPU6050 | MPU6500 |
|------|---------|---------|
| 推出年 | 2011 | 2014 |
| 軸數 | 6 軸 | 6 軸 |
| 加速度計雜訊 | 400 μg/√Hz | 300 μg/√Hz |
| 陀螺儀雜訊 | 0.005°/s/√Hz | 0.01°/s/√Hz |
| 介面 | I²C only | I²C + SPI |
| 工作電壓 | 2.375~3.46V | 1.71~3.6V |
| 工作電流 | 3.6 mA | 3.2 mA |
| Arduino 函式庫 | **超多範例** | 略少但夠用 |
| 價格 | NT$60 | NT$80~120 |

**結論**：兩者在四軸應用上**飛起來基本感覺不出差**。MPU6500 更新、雜訊稍低、SPI 比較快，但 MPU6050 的學習資源遠遠多得多。

## 真正值得考慮的升級：MPU9250

```
MPU9250 = MPU6500（6 軸） + AK8963 磁力計（3 軸）
       = 9 軸感測器
```

**為什麼磁力計重要？**

- MPU6050/6500 **沒有絕對方位**：開機那一刻為 0°，飛起來累積誤差，**Yaw 角會持續漂移**
- 磁力計提供**地磁北方**作為絕對參考，Yaw 角不會漂
- 你要做 **GPS 自動飛行**（「飛去某座標」），飛機必須知道**自己現在面朝哪個方向**才能轉向過去
- 沒磁力計 → 飛機只知道「我在這裡」「目標在那裡」，但不知道「我面朝哪邊」 → 起飛時要先手動轉一圈讓 GPS 推算航向，又慢又不準

**MPU9250 約 NT$120~150**，比 MPU6050 貴 NT$60~90，但對你的 GPS 任務**價值最高**。

## 為什麼整個系統要 3 片控制板

**無人機上其實是 2 片，地面 1 片：**

| 位置 | 板子 | 為什麼不能合併 |
|------|------|---------------|
| 無人機 | **ESP32** | 跑飛控 PID（每秒 250~1000 次），必須**獨佔 CPU** |
| 無人機 | **ESP32-CAM** | 鏡頭+WiFi 串流會吃光 CPU，且鏡頭佔用大量 GPIO（只剩 9 根可用） |
| 地面 | **Arduino Nano** | 當 USB ↔ NRF24 的橋接器（電腦沒有 NRF24 接腳，需要橋接） |

### 為什麼不能用一片 ESP32 包辦？
- 影像串流和 PID 對 CPU 搶資源，飛控時序會抖動 → **飛機會抽搐**
- ESP32-CAM 的鏡頭佔用 GPIO 0、5、18、19、21、22、23、25、26、27、32、34、35、36、39 → 飛控要用的腳幾乎都被佔光

### 為什麼地面要 Nano？
- 電腦只有 USB，不能直接控制 NRF24 模組
- Nano 負責「電腦 USB 訊號 → NRF24 無線封包」的轉換

> 你也可以用第二顆 ESP32 取代 Nano，但 Nano 你已經有了，不用花錢。

## ESP32 該買哪一款

買 **ESP32 DevKit V1（30 pin 版本）**，也叫 **ESP32-WROOM-32 DevKit**。

| 規格 | 你的需求 | 可用腳位 |
|------|---------|---------|
| 你需要的訊號腳 | 15 根 | |
| 30-pin DevKit V1 | 提供 25 個可用 GPIO | ✅ 夠用，剩 10 根可擴充 |
| 38-pin 版本 | 多 4 根 GPIO（多出 GPIO 0、9、10、11）| 可買但沒必要 |

**選購重點**：
- **CP2102 或 CH340** USB 轉換晶片（兩種都行，CP2102 驅動裝過就免裝）
- 板載 **3.3V 穩壓 800mA**（AMS1117，標配都有）
- 標 **ESP32-WROOM-32** 字樣（不要買 ESP32-S2、S3、C3 這些變種，函式庫和 GPIO 編號不一樣）

⚠️ **避雷**：別買「ESP32-CAM 開發板」當主飛控用，那是專門給鏡頭用的，GPIO 被鏡頭吃光。
